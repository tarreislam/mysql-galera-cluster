# MariaDB Galera Cluster

A Dockerized MariaDB 12 cluster with Galera 4 replication, automated configuration templating, and scheduled backups.

---

## Getting Started

### 1. Configure your environment

Copy the example env file:

```bash
cp .env.example .env
```

Open `env-calculator.html` in your browser and fill in your hardware specs (RAM, CPU cores, disk type, network speed, max latency). It will generate recommended values for the performance-related variables. Copy the output into your `.env` file.

Then fill in the remaining cluster-specific variables manually (node addresses, cluster name, SST credentials, etc.).

### 2. Start the cluster

**First node — bootstrap a new cluster:**

Set `MYSQL_ROLE=wsrep-new-cluster` in `.env`, then:

```bash
docker-compose up -d mariadb
```

**Subsequent nodes — join an existing cluster:**

Leave `MYSQL_ROLE` empty or unset in `.env`, then:

```bash
docker-compose up -d mariadb
```

## Environment Variables

### Core

| Variable | Description | Example |
|---|---|---|
| `MYSQL_ROLE` | Set to `wsrep-new-cluster` to bootstrap, leave empty to join | `wsrep-new-cluster` |
| `MYSQL_ROOT_PASSWORD` | Root password for MariaDB | `root` |

### Galera / WSREP

| Variable | Description | Example |
|---|---|---|
| `WSREP_CLUSTER_NAME` | Name shared by all cluster nodes | `MyCluster` |
| `WSREP_NODE_NAME` | Unique name for this node | `node-01` |
| `WSREP_NODE_ADDRESS` | IP address of this node | `192.168.1.246` |
| `WSREP_CLUSTER_ADDRESS` | Comma-separated IPs of all nodes | `192.168.1.10,192.168.1.11,192.168.1.12` |
| `WSREP_SST_METHOD` | State snapshot transfer method | `mariabackup` |
| `WSREP_SST_AUTH_USER` | SST user | `sst_user` |
| `WSREP_SST_AUTH_PASS` | SST password | `sst_pass` |
| `WSREP_SST_DONOR` | Preferred donor nodes | `node-02,node-03` |
| `WSREP_SLAVE_THREADS` | Parallel replication threads | `16` |
| `WSREP_MAX_WS_SIZE` | Max write-set size | `64M` |
| `WSREP_RETRY_AUTOCOMMIT` | Auto-retry on deadlock | `2` |
| `WSREP_PROVIDER_OPTIONS` | Advanced Galera tuning options | `evs.keepalive_period=PT1.0S;...` |

### Performance (generated by env-calculator.html)

| Variable | Description |
|---|---|
| `MAX_CONNECTIONS` | Maximum client connections |
| `INNODB_BUFFER_POOL_SIZE` | InnoDB buffer pool (typically ~70% of RAM) |
| `INNODB_IO_CAPACITY` / `INNODB_IO_CAPACITY_MAX` | Disk I/O capacity hints |
| `SORT_BUFFER_SIZE` | Per-session sort buffer |
| `JOIN_BUFFER_SIZE` | Per-session join buffer |
| `TMP_MEMORY_TABLE_SIZE` / `MAX_HEAP_TABLE_SIZE` | In-memory temp table limits |
| `TABLE_OPEN_CACHE` | Number of open table descriptors to cache |
| `MAX_ALLOWED_PACKET` | Max packet size for large queries/blobs |

### Backups

| Variable | Default | Description |
|---|---|---|
| `BACKUP_ENABLED` | `false` | Set to `true` to enable scheduled backups |
| `BACKUP_HOUR` | `23` | Hour of day (0–23) to run the backup |
| `BACKUP_RETENTION_DAYS` | `14` | Days to keep backup files before deletion |

Backups are written to `/backups` inside the container, which maps to `./backups` on the host. Logs are written to `/var/log/backup.log`.

To verify the cron is registered:

```bash
docker exec -it mariadb crontab -l
```

## Crash Recovery

If a node crashes and fails to rejoin, and this node was the last primary, Galera may mark it as unsafe to bootstrap. The startup script handles this automatically: when `MYSQL_ROLE=wsrep-new-cluster`, it patches `grastate.dat` to force `safe_to_bootstrap: 1` before starting.

For a joiner node, `sst_in_progress` and `galera.cache` are cleared on startup to ensure a clean SST.

---

## Debug Mode

Set `MYSQL_ROLE=debug` to start the container without launching MariaDB. The container will stay alive indefinitely, allowing you to inspect configs or run commands manually:

```bash
docker exec -it mariadb bash
```

## Volumes

| Host path | Container path | Purpose |
|---|---|---|
| `./mariadb_data` | `/var/lib/mysql` | Database data directory |
| `./backups` | `/backups` | Backup output directory |

---

## Networking

The container runs with `network_mode: host`, meaning it binds directly to the host network. This is required for Galera's inter-node communication to work correctly without extra port mapping.
